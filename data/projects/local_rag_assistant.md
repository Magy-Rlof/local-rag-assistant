# 示例项目：Local RAG Assistant

> 说明：本文档为模拟示例数据，用于说明当前项目能力。

## 项目定位

Local RAG Assistant 是一个面向求职资料管理和岗位匹配分析的本地 RAG 应用。它用于把简历、岗位描述、项目说明和行业资料集中到本地资料库中，并通过语义检索和大模型回答，帮助用户更快分析岗位匹配、项目亮点、简历优化方向和面试表达。个人复盘资料保留为本地私有内容，不作为默认 RAG 资料来源。

这个项目的定位不是纯聊天机器人，也不是算法研究项目，而是一个学习型 AI 应用开发作品集项目。它重点展示如何把 RAG 技术链路落到一个具体业务场景中。

## 解决的问题

Local RAG Assistant 主要解决以下问题：

- 求职资料分散，用户难以同时管理简历、岗位 JD、项目文档和行业资料。
- 用户面对岗位 JD 时，难以快速判断自己的项目经历和技能是否匹配岗位要求。
- 修改简历和准备面试时，用户需要反复查找资料，效率低，表达也容易不够聚焦。
- 传统关键词搜索只能找到字面匹配内容，不能很好地根据语义找出相关项目、能力和岗位要求。

项目通过本地资料库、文档切分、Embedding、Qdrant 向量检索和大模型回答，把“资料管理”和“求职分析”结合起来，让用户可以围绕自己的资料进行问答、复盘和准备。

## 核心功能

- 递归读取公开示例资料和本地私有资料
- 支持 Markdown、DOCX、PDF 等常见文档格式
- 按标题或文档结构切分文档片段
- 使用 BAAI/bge-m3 生成文本向量
- 使用 Qdrant 本地模式进行向量检索
- 使用 DeepSeek-V4-Pro 基于检索片段生成回答
- 返回引用来源和相似度分数
- 支持流式输出，前端可以实时显示模型生成过程，并在完成后使用后端最终答案校准展示内容
- 支持基础任务路由和上下文组装，用于区分项目复盘、简历修改、岗位推荐、技术解释和知识库维护等问题
- 支持回答完整性校验，对明显未完成的回答可进行一次自动续写
- 提供 Web 工作台，用于上传资料、查看索引状态、进行问答和查看引用来源
- 支持简历资料、岗位资料、项目资料和行业资料等分类管理

## 技术价值

项目展示了 RAG 的基本工程链路，包括文档加载、切分、Embedding、向量入库、语义检索、上下文拼接和模型回答。

从学习价值看，Local RAG Assistant 让开发者直接理解 RAG 的底层流程，而不是只依赖 LangChain、LlamaIndex 或 Dify 这类框架。后续如果使用框架或低代码平台，也能判断哪些步骤被封装了，哪些地方需要自定义。

从求职价值看，该项目可以作为初级 AI 应用工程师作品集 Demo，用来说明开发者具备以下能力：

- 能把大模型能力接入具体业务场景。
- 理解 RAG 的完整链路，而不仅是调用聊天接口。
- 能处理文档解析、向量检索、Prompt 拼接、API 调用和前端交互。
- 能说明本地资料管理、隐私边界和第三方 API 调用之间的关系。

## 适合写进简历的表达

基于 FastAPI、React 和 Qdrant 本地模式，实现了一个面向求职资料管理和岗位匹配分析的轻量级 RAG 问答助手。系统支持上传和管理简历、岗位 JD、项目文档和行业资料，通过 BAAI/bge-m3 完成文本向量化，使用 Qdrant 进行本地向量检索，并调用 DeepSeek-V4-Pro 根据检索片段生成回答，覆盖文档加载、切分、Embedding、检索、上下文拼接、流式问答展示和引用来源返回的完整流程。

## 面试时可以这样说

这个项目解决的是个人求职资料分散、岗位匹配分析效率低的问题。我把简历、岗位 JD、项目说明和行业资料放进本地资料库，系统会将文档切分并向量化，写入 Qdrant 本地向量库。用户提问时，系统先检索相关片段，再把片段拼接到 Prompt 中交给大模型回答。这样可以基于自己的资料分析岗位匹配、项目亮点和简历优化方向。

在交互层面，项目支持基础流式输出，前端可以实时显示模型生成过程；回答完成后再以后端最终结果校准展示内容，避免前端显示与后端最终答案不一致。

我没有一开始就用 LangChain 或 LlamaIndex，而是先手写一版 RAG 流程，目的是理解文档加载、切分、Embedding、向量检索和上下文拼接这些底层环节。后续如果使用 LangChain、LlamaIndex 或 Dify，我能更清楚它们封装了哪些能力，也知道什么时候需要自定义。

## 局限性

当前项目属于学习型和作品集 Demo，不是生产级企业系统。它暂时不适合作为多人协作平台，也没有完整的权限管理、高并发处理、审计日志和企业级部署能力。

项目中的 Qdrant 向量库和资料目录可以保存在本地，但 Embedding 和 Chat 模型调用仍可能把文本发送到第三方模型 API。因此真实敏感简历、公司资料或隐私文档在使用前应先脱敏，不能简单宣称“数据完全不出本地”或“绝对隐私安全”。

当前版本已支持基础流式输出，但仍可以继续改进引用溯源粒度、混合检索、重排序、多轮上下文管理和自动化岗位信息更新等能力。当前多轮对话主要依赖前端历史、任务路由和上下文组装，不等同于生产级会话记忆或复杂 Agent 编排。
